# Stock Incremental Game — Full Replacement One-Time Upgrade Sheet

## Purpose
This document replaces the earlier one-time upgrade list with a cleaner, more structured upgrade framework.

### Design rules used
- **Upgrades do not unlock systems** — Research does that.
- Upgrades are **one-time purchases only**.
- Upgrades should improve functionality and efficiency inside already unlocked systems.
- Upgrades should be grouped into clear categories.
- Upgrades should have clear **early / mid / late** pacing.
- Upgrades should usually stay in the **10–25% power band** unless they are a major milestone.

This sheet is intended to be the current source of truth for standard one-time upgrades.

# Balance Notes
## General strength guidance
- Early upgrades should feel immediately useful but not run-defining.
- Mid upgrades should smooth transitions into more complex systems.
- Late upgrades should reinforce established playstyles rather than unlocking brand-new systems.

## Strength guidance used
- Most upgrades stay in the **10–25%** range.
- Flat-value upgrades are used only where simple early-game readability is helpful.
- No upgrade here is intended to replace repeatable Optimisations.
- No upgrade here is intended to replace Research unlocks.

# Implementation Notes
## Recommended data shape
```ts
type UpgradeCategoryId =
  | 'tradingDesk'
  | 'research'
  | 'institutions'
  | 'automation'
  | 'infrastructure'
  | 'complianceLobbying';

type UpgradeDefinition = {
  id: string;
  category: UpgradeCategoryId;
  name: string;
  description: string;
  cost: number;
  unlockConditionDescription: string;
  isUnlocked: (state: GameState) => boolean;
  isPurchased: (state: GameState) => boolean;
};
```

## Important design rule
Unlock conditions should only check whether the **relevant system or unit already exists**.
They should **never unlock a new system** by themselves.
